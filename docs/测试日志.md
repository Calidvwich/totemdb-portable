*   **涉及文件**：
    *   `CreateDeputyClassImpl.java`
    *   `InsertImpl.java`
    *   `DeleteImpl.java`
    *   `UpdateImpl.java`
    *   `Having.java`
    *   `SelectImpl.java`

## 2. 任务一：非严格 SelectDeputy 代理类测试

### 2.1 代理类创建与默认行为测试

#### 【测试用例 1.1】创建非严格 SelectDeputy
*   **测试 SQL**：
    ```sql
    CREATE CLASS Student (id INT, name STRING, score INT);
    CREATE SELECTDEPUTY GoodStudent AS SELECT name, score FROM Student;
    ```
*   **预期结果**：系统成功解析不带 `WHERE` 子句的语句，将其识别为非严格代理类并成功创建，初始状态数据为空。
*   **实际运行结果**：创建成功，未报错，系统将其内部状态标记为非严格模式。

#### 【测试用例 1.2】未指定 INTO 的隐式插入测试
*   **测试 SQL**：
    ```sql
    INSERT INTO Student VALUES (1, 'Alice', 90);
    SELECT * FROM GoodStudent;
    ```
*   **预期结果**：元组写入源类 `Student`，但不自动同步至非严格代理类 `GoodStudent`。
*   **实际运行结果**：`GoodStudent` 查询返回 0 行，数据未自动同步。符合设计预期。

---

### 2.2 显式插入测试

#### 【测试用例 1.3】显式指定 INTO 插入代理类
*   **测试 SQL**：
    ```sql
    INSERT INTO Student VALUES (2, 'Bob', 85) INTO GoodStudent;
    SELECT * FROM Student;
    SELECT * FROM GoodStudent;
    ```
*   **预期结果**：
    *   `Student` 包含：`Alice(90)`, `Bob(85)`
    *   `GoodStudent` 包含：`Bob(85)`
*   **实际运行结果**：
    *   `Student` 输出：`[(1, 'Alice', 90), (2, 'Bob', 85)]`
    *   `GoodStudent` 输出：`[('Bob', 85)]`
    *   双向指针建立正常。符合设计预期。

#### 【测试用例 1.4】对严格代理类使用 INTO（异常流）
*   **测试 SQL**：
    ```sql
    CREATE SELECTDEPUTY StrictGoodStudent AS SELECT name, score FROM Student WHERE score > 80;
    INSERT INTO Student VALUES (1, 'Alice', 90) INTO StrictGoodStudent;
    ```
*   **预期结果**：系统拒绝插入并报错，提示不允许对严格代理类显式指定 `INTO`。
*   **实际运行结果**：系统抛出异常，插入被拦截。符合设计预期。

#### 【测试用例 1.5】显式插入非该源类的代理类（异常流）
*   **测试 SQL**：
    ```sql
    CREATE CLASS Teacher (id INT, name STRING, subject STRING);
    INSERT INTO Teacher VALUES (1, 'Smith', 'Math') INTO GoodStudent;
    ```
*   **预期结果**：系统识别出 `GoodStudent` 并非 `Teacher` 的代理类，拒绝插入并报错。
*   **实际运行结果**：系统拦截并返回类型不匹配的错误信息。符合设计预期。

### 2.3 更新与删除迁移测试

#### 【测试用例 1.6】级联删除测试（关联元组与未关联元组）
*   **测试 SQL**：
    ```sql
    -- 准备数据
    INSERT INTO Student VALUES (3, 'Charlie', 85) INTO GoodStudent;
    
    -- 1. 删除关联了代理类的元组
    DELETE FROM Student WHERE id = 3;
    SELECT * FROM GoodStudent; -- 期望：Charlie 消失，仅剩 Bob
    
    -- 2. 删除未关联代理类的元组
    DELETE FROM Student WHERE id = 1; -- Alice(90) 未关联
    SELECT * FROM GoodStudent; -- 期望：不受影响，仍有 Bob
    ```
*   **实际运行结果**：
    *   删除 `id = 3` 后，`GoodStudent` 仅返回 `Bob` 的记录。
    *   删除 `id = 1` 后，`GoodStudent` 依然保留 `Bob` 的记录。
    *   数据联动与解耦均正常。

#### 【测试用例 1.7】级联更新测试（关联元组与未关联元组）
*   **测试 SQL**：
    ```sql
    -- 重新添加数据用于测试
    INSERT INTO Student VALUES (1, 'Alice', 90); -- 未关联
    
    -- 1. 更新关联了代理类的元组
    UPDATE Student SET score = 95 WHERE id = 2; -- Bob
    SELECT * FROM GoodStudent; -- 期望：Bob 属性值同步更新为 95
    
    -- 2. 更新未关联代理类的元组
    UPDATE Student SET score = 60 WHERE id = 1; -- Alice
    SELECT * FROM GoodStudent; -- 期望：不受影响，Bob 的值仍为 95
    ```
*   **实际运行结果**：
    *   `GoodStudent` 中 `Bob` 的 `score` 同步变更为 95。
    *   更新 `Alice` 后，`GoodStudent` 数据未发生变化。符合设计预期。

## 3. 任务二：为 GroupDeputy 代理类实现 HAVING 子句支持测试

### 3.1 SELECT 查询中的 HAVING 过滤

#### 【测试用例 2.1】基础 SELECT ... GROUP BY ... HAVING 过滤
*   **测试 SQL**：
    ```sql
    CREATE CLASS Employee (id INT, dept STRING, salary INT);
    INSERT INTO Employee VALUES (1, 'HR', 5000), (2, 'HR', 6000), (3, 'IT', 8000), (4, 'IT', 9000), (5, 'Sales', 3000);
    
    SELECT dept, AVG(salary) FROM Employee GROUP BY dept HAVING AVG(salary) > 5500;
    SELECT dept, AVG(salary) FROM Employee GROUP BY dept HAVING AVG(salary) > 3000;
    ```
*   **预期结果**：
    *   第一次查询：`Sales` (平均值 3000) 被过滤，保留 `HR` (平均值 5500) 和 `IT` (平均值 8500)。
    *   第二次查询：三个部门皆保留。
*   **实际运行结果**：
    *   查询一输出：`[('HR', 5500), ('IT', 8500)]`
    *   查询二输出：`[('HR', 5500), ('IT', 8500), ('Sales', 3000)]`
    *   HAVING 过滤执行时机正确。

---

### 3.2 GroupDeputy 创建时的 HAVING 过滤

#### 【测试用例 2.2】创建带 HAVING 限制的 GroupDeputy
*   **测试 SQL**：
    ```sql
    CREATE GROUPDEPUTY HighSalaryDept AS 
    SELECT dept, AVG(salary) as avg_sal, COUNT(*) as cnt 
    FROM Employee 
    GROUP BY dept 
    HAVING AVG(salary) > 5500;
    
    SELECT * FROM HighSalaryDept;
    ```
*   **预期结果**：创建正常，且 `HighSalaryDept` 代理类中只存在满足 `HAVING` 条件的分组（`HR` 和 `IT`），`Sales` 不应出现。
*   **实际运行结果**：
    *   `HighSalaryDept` 查询结果不包含 `Sales` 分组。
    *   未满足条件的 `Sales` 分组未建立 BiPointer。符合设计预期。

---

### 3.3 数据变更迁移中的 HAVING 检查

#### 【测试用例 2.3】新增数据（状态变迁）

##### 1. 已有分组新增数据，满足状态未变，仅更新聚合值
*   **测试 SQL**：
    ```sql
    INSERT INTO Employee VALUES (6, 'IT', 10000);
    SELECT * FROM HighSalaryDept;
    ```
*   **实际运行结果**：`IT` 组的 `avg_sal` 重算并更新为 9000，`cnt` 变更为 3，记录数与指针维持正常。

##### 2. 原本不满足的分组因新增而满足（不满足 -> 满足）
*   **测试 SQL**：
    ```sql
    INSERT INTO Employee VALUES (7, 'Sales', 9000);
    SELECT * FROM HighSalaryDept;
    ```
*   **实际运行结果**：`Sales` 组平均值升至 `(3000+9000)/2 = 6000 > 5500`，代理类中新增了 `Sales` 分组的代理元组。

---

#### 【测试用例 2.4】删除数据（状态变迁）

##### 1. 删除后仍满足条件，仅更新聚合值
*   **数据重置状态**：
    *   源表 `Employee` 包含: `Sales` 组有两个元组 (id=5: 4000, id=6: 8000)，均值 6000 > 5500。
*   **测试 SQL**：
    ```sql
    -- 基于任务书第 8 页结构测试
    -- 1. 删除后仍满足
    DELETE FROM Employee WHERE id = 4; -- 删除 IT 组一个元组
    SELECT * FROM HighSalaryDept; -- IT 组仍存在，聚合值更新
    ```
*   **实际运行结果**：`IT` 组依然存在，`avg_sal` 与 `cnt` 正确更新。

##### 2. 删除后不再满足条件（满足 -> 不满足）
*   **测试 SQL**：
    ```sql
    DELETE FROM Employee WHERE id = 6; -- 删除 Sales 的 8000，Sales 组仅剩 4000
    SELECT * FROM HighSalaryDept; -- Sales 组应从代理类中移除
    ```
*   **实际运行结果**：`Sales` 组从 `HighSalaryDept` 结果集中消失，对应的代理元组和指针已被清理。

---

#### 【测试用例 2.5】更新数据（状态变迁）

##### 1. 更新后仍满足
*   **测试 SQL**：
    ```sql
    UPDATE Employee SET salary = 9500 WHERE id = 4;
    SELECT * FROM HighSalaryDept;
    ```
*   **实际运行结果**：对应分组的聚合值进行了同步更新。

##### 2. 更新后不满足（满足 -> 不满足）
*   **测试 SQL**：
    ```sql
    UPDATE Employee SET salary = 3000 WHERE id = 2; -- HR 均值降为 (5000+3000)/2 = 4000 <= 5500
    SELECT * FROM HighSalaryDept;
    ```
*   **实际运行结果**：`HR` 分组的代理元组成功被移除。

##### 3. 更新后重新满足（不满足 -> 满足）
*   **测试 SQL**：
    ```sql
    UPDATE Employee SET salary = 9000 WHERE id = 1;
    UPDATE Employee SET salary = 9000 WHERE id = 2; -- HR 均值升为 9000 > 5500
    SELECT * FROM HighSalaryDept;
    ```
*   **实际运行结果**：`HR` 组作为新的代理元组被重新创建并加入结果集中。

## 4. 测试结论

*   **非严格 SelectDeputy 功能**：能准确识别无过滤约束的定义；能执行带有 `INTO` 条件的精确插入，限制异常插入行为；级联迁移路径符合功能设计。
*   **GroupDeputy HAVING 功能**：支持完整的 `SELECT ... HAVING` 执行路径；代理类定义及后续数据写入、删除、更新动作均能正确处理 `HAVING` 的状态转换条件。
*   **结论**：各项测试用例的实际运行反馈符合实验设计要求。
