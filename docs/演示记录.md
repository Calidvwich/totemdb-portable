## 任务二

已在 Android 前端执行：

- `HAVING AVG(salary) > 5000`：保留 HR、IT。
- `HAVING AVG(salary) >= 3000`：保留 HR、IT、Sales。
- 创建 HighSalaryDept 时过滤 Sales。
- INSERT IT 10000：IT 更新为 avg=9000、cnt=3。
- INSERT Sales 9000：Sales 以 avg=6000、cnt=2 进入。
- DELETE IT id=4：IT 更新后仍保留。
- DELETE Sales 高工资成员：Sales 退出。
- UPDATE IT id=4 为 9500：IT 更新为 avg=8750。
- UPDATE HR id=2 为 3000：HR 退出。
- 将 HR 两名员工更新为 9000：HR 重新进入。

## 补充验收 SQL 与执行说明

补充验收脚本见 [验收SQL.md](验收SQL.md)。该文档用于在 Android 前端 SQL 输入框中逐条复制执行，补充截图之外的完整验收路径和边界观察点。

建议执行顺序：

1. 先执行 `验收SQL.md` 第 1-3 节，验收任务一非严格 SelectDeputy。
2. 再执行第 4-5 节，验收任务二 GroupDeputy HAVING。
3. 最后执行第 6 节，观察边界场景和已知风险。

章节对应关系：

- 第 1 节：非严格 SelectDeputy 基础成功路径，包括创建为空、普通 INSERT 不传播、显式 `INTO` 插入。
- 第 2 节：错误场景，包括严格代理拒绝显式 `INTO`、非同源代理拒绝写入，以及失败后数据不污染。
- 第 3 节：非严格 SelectDeputy 的 DELETE / UPDATE 迁移。
- 第 4 节：普通 SELECT HAVING、GroupDeputy HAVING 创建过滤、无 HAVING 回归。
- 第 5 节：GroupDeputy 在 INSERT / DELETE / UPDATE 后的 HAVING 迁移。
- 第 6 节：边界场景，包括 `>` 与 `>=`、`COUNT(*)`、HAVING 别名，以及需要进一步验证的非主路径风险。

一致性观察方式：

- 以 `SELECT * FROM 源类` 和 `SELECT * FROM 代理类` 的结果为主要验收依据。
- 可选执行 `showb` 或 `show BiPointerTable` 观察 BiPointerTable。满足代理条件或显式进入代理的源元组应有对应指针；被 HAVING 过滤、已删除或未显式进入非严格代理的源元组不应留下指向代理类的指针。
- SwitchingTable 不直接作为主要演示输出，但可通过代理类属性是否按源属性正确同步来间接验证映射有效。

说明：PDF 任务书中的部分 SQL 示例可能存在笔误或阈值差异，具体演示 SQL 以当前仓库的 `README.md`、`todo.md` 和 `docs/验收SQL.md` 为准；任务要求本身不变。
